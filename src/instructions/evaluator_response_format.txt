Response format:
Return exactly one valid JSON object and no markdown, code fences, or surrounding prose.
Schema: {"answer":"<single-line answer>","evidence":"<free-form evidence citing supporting files or code with backticked project-relative refs>","scope":["<normalized repository-relative path>"]}
The `answer` field is only the exact answer token/string: `yes` or `no` for yes/no questions, `idk`, `malformed`, an option letter, or another exact single-line answer when the question asks for one. Do not put evidence or explanation in `answer`. When `evidence` cites project files, directories, or code locations, format those references as project-relative refs enclosed in backticks, such as `src/lib.rs` or `src/lib.rs:42`; do not cite absolute paths. `scope` is the minimal subset of the visible staged Git-tracked snapshot whose unchanged contents make the correct answer invariant to changes outside it; it is not the list of evidence citations. Propose a narrower scope only when that narrower scope is sufficient to determine the correct answer among all valid answers. Non-`idk` answers must keep `scope` inside the enforced scope. If the smallest sufficient scope would be ["."] or any path outside the enforced scope, answer `idk` and keep `scope` inside the enforced scope so canon can retry with wider access. Use ["."] when the answer depends on project-wide absence, consistency, duplication, garbage, overall quality, or denied/inaccessible paths. If the enforced task `scope` is narrower than ["."] and the question requires repository-wide or cross-module evidence, answer `idk` instead of drawing a positive or negative conclusion from incomplete context. If visible code says the relevant behavior lives outside the enforced scope, answer `idk` rather than failing the project from that incomplete slice. Never include denied or inaccessible paths in `scope`. Denied paths are intentionally outside the allowed evidence boundary; do not answer `idk` solely because a denied path is unreadable. The current project state is the staged Git snapshot exposed at the working directory; do not treat files that exist only in `HEAD`, cache history, or diagnostic logs as current project files. A path that appears only as an already-staged deletion is absent from the current staged project state, so do not cite that removed path as an obsolete current file or code path. In `git ls-files -s` output, the trailing numeric column is the Git index stage, not a byte size; use file content, `wc`, or `stat` if file size matters. Evaluator agents are read-only; do not answer `idk` merely because a build or test command cannot run in the sandbox. Use visible staged source and test files as evidence unless the question specifically asks whether a command executed successfully. A reusable cache hit is not an evaluator interrogation; questions about every evaluator interrogation concern only turns where `canon check` actually asks the evaluator model.
